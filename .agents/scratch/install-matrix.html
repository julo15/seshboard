ARCH  AppDelegate is now the canonical install orchestrator

Every .app launch reads ~/Library/Application Support/Seshctl/installed-v1.json and compares three signals against the running bundle:

  • bundlePath ≠ current Bundle.main.bundleURL (bundle moved)
  • version ≠ current CFBundleShortVersionString (release upgrade)
  • marker.installedAt < bundle's Contents/MacOS/SeshctlApp mtime (any cp / drag-over replace)

Any mismatch → silent FirstLaunchInstaller.install(bundleURL:). No welcome panel, just a refresh of hook scripts + settings entries + marker. End-user upgrades and make install both flow through this — single code path.

CLI surface shrank to two zero-flag verbs (plus one option):

  • seshctl install — full install (CLI symlinks, hook scripts with defensive guard, settings entries, marker). Used as a manual recovery path; AppDelegate handles the normal case.
  • seshctl uninstall [--delete-history] — full uninstall. Clears codex_hooks = true always; with the flag, also removes the session DB.

What's actually left on the user's Mac after a "full" uninstall

Artifact Status Why

TL;DR All six scenarios resolve to ✅ OK end states. The CLI surface is two verbs (seshctl install / seshctl uninstall [--delete-history]). Both end-user upgrade and developer iteration flow through AppDelegate's launch-time reconciler, so hooks always stay in sync with the deployed bundle without any manual step. The remaining residue after a full uninstall is the session DB (opt-in to remove) and macOS-controlled TCC grants (no API).
