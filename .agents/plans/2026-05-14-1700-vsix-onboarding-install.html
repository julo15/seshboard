Seshctl — Editor Extension Onboarding

How DMG users get the VS Code / Cursor companion extension installed, and how it stays current.

Companion to the plan. This canvas is a visual summary of 2026-05-14-1700-vsix-onboarding-install.md in the same directory — that markdown file is the source of truth. If the two disagree, trust the .md. Regenerate this HTML by asking Claude to "rebuild the canvas from the plan" when the plan changes substantively.
One-line summary Bundle the .vsix inside Seshctl.app. On fresh install, open an Editor Integrations window that lists each detected editor and lets the user install per-row. On every launch, silently refresh any editor that already has Seshctl installed at a stale version.

4 — What the window actually looks like

SwiftUI view hosted by NSWindowController, ~500×420, non-resizable. Three rows = three editors I detected on this Mac. Each row's button text reflects the live status.

Auto-open trigger: only on fresh install (after the existing NSAlert). Otherwise reachable via Settings → Editor Integrations → Configure…
Hidden rows: editors not installed on this Mac don't appear at all. If none of the three are present, the window is skipped entirely on fresh install.
Failures: row stays in place; a one-line truncated stderr appears under the title, with a "Details" disclosure for the full output.

5 — Where state lives (and where it doesn't)

A deliberate choice: read from the authoritative source (the editor itself) rather than caching install state in our own marker file.

Bundled extension version (read-only, ships in DMG)

Seshctl.app/Contents/Resources/extensions/seshctl.vsix.version

One line, e.g. 0.2.1. Source of truth for "what version are we shipping". Avoids parsing the .vsix zip at runtime.

Installed extension version (live, owned by editor)

~/.vscode/extensions/julo15.seshctl-<version>/
~/.cursor/extensions/julo15.seshctl-<version>/

Read via <editor> --list-extensions --show-versions on every launch. Always-correct; survives user manually uninstalling inside VS Code.

Not introduced: Seshctl-side extension marker

~/Library/Application Support/Seshctl/extensions.json

Deliberately not added. A local cache would drift when the user uninstalls the extension inside the editor. Querying --list-extensions is cheap (~200 ms) and avoids the coherence trap.

Not introduced: opt-in flag per editor

(no file)

"Did the user previously install for this editor?" is inferred from the editor: if julo15.seshctl is present, they opted in. Silent refresh only touches those editors.
